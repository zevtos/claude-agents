# Building an AI refactoring agent: clone detection, safety, and cross-language analysis

**A practical AI refactoring agent for Claude Code should combine tree-sitter-based AST parsing, multi-strategy clone detection, and a layered safety verification stack — not formal verification, but test-based heuristics, mutation testing, and type-checking guards.** Recent research demonstrates that multi-agent architectures (MANTRA, RefAgent) achieve **82–90% success rates** on automated refactoring tasks, compared to just 8.7% for naive LLM prompting, but only when paired with static analysis to filter the **76.3% hallucination rate** in raw LLM refactoring suggestions. The core pipeline — scan, detect, propose, validate, apply, verify — must be language-aware at the edges but language-agnostic in the middle, leveraging tree-sitter's universal parsing layer and tools like ast-grep for structural pattern matching across Go, Python, Swift, Java, and TypeScript.

## Clone taxonomy determines which detection strategy to deploy

Clone detection operates across four well-defined types, each demanding fundamentally different algorithmic approaches. **Type 1 clones** (exact copies modulo whitespace and comments) yield to trivial text normalization. **Type 2 clones** (renamed identifiers and literals but identical structure) require parameterized matching — Baker's p-match algorithm constructs parameterized suffix trees using "predecessor strings" that encode the distance to the previous occurrence of each symbol, enabling linear-time detection. **Type 3 clones** (near-miss, with added/deleted/modified statements) need gap-tolerant approaches with configurable similarity thresholds. **Type 4 clones** (semantically equivalent but syntactically different implementations, such as iterative vs. recursive GCD) remain the research frontier — traditional syntactic methods fail entirely, and even the best neural models achieve only **~23% recall** on Type-4 pairs in BigCloneBench.

The practical detection toolkit spans five algorithmic families. Token-based tools like **CCFinder** tokenize source code, replace all identifiers with a generic parameter token, then use suffix-tree matching to find duplicated subsequences above a length threshold. **SourcererCC** scales further by treating each function as a bag of tokens and building an inverted index with sub-block overlap filtering — it processed **250 million lines of code** on a single workstation, making it the most scalable near-miss detector available. **NiCad** takes a different path: it uses TXL transformations to pretty-print and normalize code into a canonical form, then compares fragments by their Unique Percentage of Items (differing lines), which naturally tolerates small gaps. **PMD CPD** applies Karp-Rabin hashing on token sequences and supports 33+ languages out of the box, making it the most accessible starting point for a polyglot agent.

AST-based approaches trade scalability for structural precision. **DECKARD** represents each subtree as a characteristic vector counting occurrences of each node type, then clusters similar vectors using locality-sensitive hashing — achieving **93% accuracy** on manual inspection while scaling to millions of lines. The anti-unification technique (least general generalization) computes the most specific template that subsumes two AST fragments, replacing divergent subtrees with variables — this directly produces the parameterized pattern needed for extract-function refactoring. CloneDR's subtree hashing approach computes bottom-up structural hashes, bins identical subtrees, and only performs detailed comparison within each bucket.

## Neural embeddings catch semantic clones but struggle at scale

ML-based approaches represent code as learned vectors and measure clone similarity via cosine distance. **code2vec** decomposes methods into AST path triples (source terminal → path of node types → target terminal) and aggregates them with an attention mechanism. **CodeBERT** fine-tunes a RoBERTa transformer on source tokens, reaching **F1=0.965** on BigCloneBench. **GraphCodeBERT** extends this with data-flow edges representing where-the-value-comes-from relationships, pushing to **F1=0.971**. **ASTNN** splits ASTs into statement subtrees, encodes each with a recursive neural network, then feeds the sequence through a bidirectional GRU — preserving control-flow order that pure bag-of-tokens approaches lose.

However, these numbers require careful interpretation. A generalizability study found that CodeBERT's F1 drops significantly when evaluated on code snippets and functionality IDs not seen during training, suggesting overfitting to BigCloneBench's training split. For a practical agent, the recommended approach is a **tiered detection pipeline**: use SourcererCC or PMD CPD for fast, scalable Type 1–3 detection across the codebase, then apply neural models (GraphCodeBERT or UniXcoder) only on candidate pairs for Type-4 semantic similarity scoring. InferCode's self-supervised approach — predicting subtrees within ASTs without manual labels — is particularly promising for cross-language scenarios since it can be trained on any language's ASTs.

## The safety stack: five layers from type checks to mutation testing

Verifying refactoring correctness without formal methods requires stacking complementary heuristics, each catching different failure classes.

**Type checking forms the first, cheapest layer.** TypeScript strict mode (`strictNullChecks`, `strictFunctionTypes`, `noImplicitAny`) catches signature mismatches, null-safety violations, and property initialization errors after refactoring. Python's mypy in strict mode (`--strict`) catches incompatible return types and `NoneType` access, with reports claiming **50–65% production bug reduction**. Go's compiler catches interface satisfaction failures immediately. The key limitation: type holes (`any` in TypeScript, `# type: ignore` in Python, type erasure in Java generics) create blind spots that require supplementary checks.

**Running existing tests forms the second layer** — Fowler's classic "small transformation → run tests → continue or revert" cycle. But code coverage alone is insufficient: 100% line coverage only means each line executed once, not that boundary conditions were tested. This is where **mutation testing becomes critical as the third layer**. PIT (Java), mutmut (Python), and go-mutesting (Go) introduce small faults (changing `>` to `>=`, negating conditionals, removing void method calls) and verify that tests detect them. The MeteoR tool automates pre/post comparison: run mutation testing before refactoring, record which mutants are killed, apply the refactoring, run mutation testing again — any drop in mutation score pinpoints where behavior diverged. Teams report **"extreme confidence"** with 90%+ mutation scores on refactored regions.

**Property-based testing as the fourth layer** provides the strongest behavioral equivalence guarantee: define the property `for_all(inputs, original(x) == refactored(x))` and let Hypothesis (Python), QuickCheck (Haskell), or fast-check (TypeScript) generate thousands of inputs automatically. Properties survive refactoring by design — a **50% reduction in test updates during refactoring** compared to example-based tests. This approach is especially powerful for extract-function: use the original inline code as an oracle against the extracted function.

**AST-level diffing as the fifth layer** verifies that only intended structural changes occurred. GumTree computes edit scripts with insert, delete, update, and move actions between two ASTs, though it generates inaccurate mappings for **20–29%** of file revisions. Difftastic (Rust, tree-sitter-based) provides a practical alternative — its `--check` mode returns exit code 0 if no syntactic changes exist, making it CI-ready for verifying that formatting-only changes are truly cosmetic.

## Refactoring failure modes every agent must handle

The most dangerous refactoring bugs are subtle behavioral changes that pass type checks and most tests. **Changed evaluation order** is the most common: extracting the right operand of `&&` or `||` into a variable forces its evaluation regardless of short-circuit semantics. If `b()` has side effects — I/O, exceptions, state mutation — the refactored code behaves differently despite identical structure.

**Broken closures** manifest differently per language. JavaScript's `var` is function-scoped, so closures in loops capture the same variable (all callbacks see the final value). Python's late-binding closures mean `[lambda: i for i in range(5)]` produces five functions that all return 4. Go closures capture variables by reference. Swift's value-type closures copy structs. An AI agent extracting code containing closures must analyze capture semantics for the target language and insert appropriate fixes (copying to a local variable, using default parameter binding in Python, or explicit `[capture]` lists in Swift/C++).

**Reference vs. value semantics** create silent bugs. Python's mutable default arguments (`def f(x, to=[])`) are evaluated once at definition time — the list is shared across calls. Go slices share underlying arrays, so passing a slice to an extracted function and appending can unexpectedly modify the caller's data. Swift structs are value types copied on assignment; extracting code that modifies a struct requires `inout` parameters to maintain mutation semantics. Java's autoboxing introduces `null` where primitives were expected, causing NPEs after extracting to a method with `Integer` parameter type.

**Exception scope changes** after extraction alter error propagation: code that was inside a `try/catch` block gets extracted to a method that may need its own error handling or must declare `throws`. Go's idiomatic `(result, error)` pattern means every extracted function must return an error, and every call site must handle it. TypeScript's `async` functions must propagate `async` upward through the call chain. These cross-cutting concerns mean a refactoring agent needs language-specific safety analyzers even when the detection logic is language-agnostic.

## How to architect a multi-agent refactoring system

The state-of-the-art architecture follows MANTRA's two-layer multi-agent design: a **Developer Agent** extracts repository structure and applies refactorings, a **Reviewer Agent** evaluates and provides feedback, and a **Repair Agent** fixes compilation and test failures through verbal reinforcement learning. Removing any component drops performance by **40–61%**, with the Reviewer Agent contributing most. RefAgent's variant achieves a **90% median unit test pass rate** and reduces code smells by 52.5%, with a **+64.7% unit test pass rate** compared to single-agent approaches.

The practical pipeline for Claude Code integration looks like this: tree-sitter parses the codebase into CSTs across all target languages, with ast-grep (Rust-native, 10k+ GitHub stars) providing structural search and replace via pattern matching with `$METAVAR` wildcards. The agent scans for code smells using a combination of metric-based heuristics (cyclomatic complexity, function length, parameter count) and clone detection (SourcererCC-style token overlap for speed, followed by AST subtree hashing for precision). Detected clone groups are ranked by a confidence score incorporating test coverage of the affected region, clone similarity score, number of dependents (blast radius), and whether the code contains side effects or concurrency primitives.

**Confidence scoring** should weight multiple signals. EM-Assist's frequency-based ranking — lines that appear across multiple LLM generation attempts rank higher — provides a simple but effective proxy for extraction-point confidence. CodeScene's hotspot analysis combines code health with change frequency to prioritize refactoring targets by ROI. The recommended composite score weights test coverage (30%), structural similarity (25%), type-safety level of the target language (20%), scope complexity (15%), and side-effect analysis (10%).

For verification, the agent should follow a **characterization-test-first** discipline: capture current behavior as golden snapshots before any refactoring, then compare outputs against them after transformation. Go's `testdata/` golden file convention, Jest snapshots, and Approval Tests (available for Java, Python, Go, Ruby) all serve this purpose. The hard rule: no characterization tests means no AI refactoring on that region. Each refactoring should be an atomic git commit with tests run between commits, enabling `git revert` of individual changes. For high-risk refactorings, the Strangler Fig pattern — routing traffic between old and new implementations via feature flags — provides runtime safety that compile-time checks cannot.

## Tree-sitter enables universal parsing but not universal semantics

Tree-sitter provides a consistent parsing interface across 200+ languages, producing concrete syntax trees with error recovery and incremental parsing. Every grammar exposes named nodes (corresponding to grammar rules), fields (semantic relationships like `condition`, `body`), and supertypes (abstract categories like `expression`, `statement`). GitHub uses tree-sitter for code navigation across 100+ languages, and Copilot uses it for prompt extraction.

The critical challenge is that **tree-sitter grammars have no standardized vocabulary**. An analysis of 5,000+ node types across 25 languages found that grammar authors define names independently. A function definition is `function_declaration` in Go, `function_definition` in Python, `method_declaration` in Java, and `function_declaration` in TypeScript. Variable declarations range from Go's `short_var_declaration` to Python's `assignment` (Python has no declaration syntax) to Java's `local_variable_declaration`. Building cross-language analysis requires either per-language query files (GitHub's approach) or an abstraction layer mapping language-specific patterns to generic concepts.

Three tools implement this abstraction effectively:

- **ast-grep** provides pattern-based code search and rewrite using tree-sitter CSTs with `$METAVAR` wildcards, supporting 35+ languages with a YAML rule configuration system for custom linting and refactoring rules
- **Uber's PolyglotPiranha** uses match-replace graphs on tree-sitter trees for automated code refactoring across 10 languages (Java, Kotlin, Swift, Go, Python, TypeScript), specifically designed for large-scale codebase maintenance like feature-flag cleanup
- **Semgrep** uses tree-sitter internally for generic pattern matching across 30+ languages, with the largest ecosystem of pre-built security and quality rules

For cross-language clone detection specifically, the most promising recent approaches bypass AST normalization entirely. **SLACC** (ICSE 2020) uses dynamic analysis — executing functions on generated inputs and clustering by I/O behavior — achieving **94.1% precision** on Java-Python clone pairs without any common intermediate representation. **C4** (ICPC 2022) applies contrastive learning on CodeBERT embeddings, reaching **F1=0.92** on cross-language pairs. These embedding-based approaches scale better than IR-based approaches because they don't require compilation or bytecode generation for interpreted languages.

## Cross-language scoping and semantics demand language-specific safety checks

The five target languages differ profoundly in scoping rules, and these differences directly impact refactoring safety. Python's LEGB rule and absence of block scoping mean that `if`/`for`/`while` blocks don't create new scopes — extracting code from inside a loop may inadvertently expose variables that were "local" only by convention. Go's short variable declarations (`:=`) can shadow outer variables silently, and the `go vet` shadow analyzer is the primary defense. TypeScript's `var` hoisting creates function-scoped variables that behave differently from `let`/`const` block-scoped ones — extracting code that mixes these requires careful binding analysis. Swift's five access control levels (`open` > `public` > `internal` > `fileprivate` > `private`) interact with value-type closure capture in ways that Java's simpler model doesn't.

The theoretical framework for language-independent name resolution exists: **scope graphs** (Néron et al., ESOP 2015) model declarations, references, and scopes as a graph where name resolution becomes path-finding. The associated Statix meta-language can specify type systems using scope graphs, demonstrated for Java-like languages. However, for a practical agent, the more immediate approach is to build language-specific safety analyzers that plug into a common refactoring framework. Each analyzer implements free-variable analysis (identifying variables referenced but not declared in the extracted region), modified-variable analysis (which free variables are written to), and control-flow analysis (return statements, break/continue that cross extraction boundaries).

Error handling patterns diverge most dramatically: Go's explicit `(result, error)` returns, Python's exception propagation, Swift's marked `throws` functions, Java's checked exceptions, and TypeScript's untyped `catch` clauses each impose different constraints on extract-function. An extracted Go function must return an error; an extracted Java method must declare every checked exception; an extracted Swift function must be marked `throws` if it contains any throwing call. A practical agent should maintain a per-language error-handling policy that automatically adjusts function signatures during extraction.

## Conclusion

Building a practical AI refactoring agent requires combining three capabilities that don't exist well in any single tool today. First, **multi-strategy clone detection** that uses fast token-based scanning (SourcererCC/PMD CPD) for breadth and neural embeddings (GraphCodeBERT/UniXcoder) for semantic depth, with anti-unification for template extraction from clone groups. Second, a **layered safety stack** — type checking, existing tests, mutation testing on the refactored region, property-based equivalence testing, and AST diffing — where each layer catches failures the previous one misses. Third, **language-specific safety analyzers** that handle the profound differences in scoping, error handling, value/reference semantics, and closure capture across Go, Python, Swift, Java, and TypeScript.

The most actionable insight from recent research is that **multi-agent architectures with reviewer feedback loops vastly outperform single-pass refactoring**. The combination of LLM-generated refactoring proposals with static-analysis filtering (removing the 76% hallucination rate) and test-based verification produces industrially viable results. Tree-sitter with ast-grep provides the universal parsing substrate, but the semantic layer must remain language-aware. The key tools to integrate are ast-grep for structural pattern matching, PolyglotPiranha for large-scale automated transforms, PIT/mutmut for mutation-testing-based safety verification, and Difftastic for AST-level change auditing. Start with the highest-confidence refactorings (exact clone deduplication with high test coverage) and expand scope as the safety stack proves reliable.
